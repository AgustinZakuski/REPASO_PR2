Patrones de diseño — tocá cada uno

Singleton▼

Garantiza que exista una única instancia de un objeto. Se implementa con constructor privado, atributo estático y privado, y un método estático y público para obtener la instancia. Nivel de uso: muy alto.

Factory Method▼

Define una interfaz estándar para crear objetos de una familia, sin exponer la clase concreta que se instancia. Nivel de uso: alto. Tiene una variante llamada Simple Factory.

Abstract Factory▼

Centraliza la fabricación de objetos de distintas familias en un solo lugar. Nivel de uso: bajo, visto como introducción en la materia.

Adapter▼

Adapta un objeto o interfaz para que pueda ser usado por otro objeto que, de otro modo, no podría usarlo. Nivel de uso: muy alto.

Composite▼

Permite tratar a objetos simples y a objetos compuestos de la misma manera, simplificando el código que los usa. Nivel de uso: medio.

Proxy▼

Provee un sustituto de otro objeto y controla el acceso a él, permitiendo hacer algo antes o después de que la solicitud llegue al objeto original.

State▼

Se usa cuando un objeto necesita modificar su comportamiento cada vez que cambia su estado interno. Nivel de uso: alto.

Strategy▼

Permite disponer de varios métodos o algoritmos para resolver un problema, y elegir cuál usar en tiempo de ejecución. Nivel de uso: alto.

Observer▼

Define una dependencia entre un sujeto y muchos observadores: cuando el sujeto cambia de estado, todos los observadores se actualizan automáticamente. Nivel de uso: alto.

Command▼

Encapsula una operación dentro de un objeto, permitiendo ejecutarla sin que quien la invoca conozca su contenido. Nivel de uso: medio.

Creacionales Estructurales Comportamiento

SOLID — tocá cada letra

S Single Responsibility ▼

Una clase debe tener una única razón para cambiar, es decir, debe hacer una sola cosa.

O Open/Closed ▼

Las entidades de software deben estar abiertas para ser extendidas, pero cerradas para ser modificadas.

L Liskov Substitution ▼

Un método que usa una referencia a la clase base debe poder usar objetos de sus clases derivadas sin necesidad de conocerlas: una subclase debe poder reemplazar a su superclase.

I Interface Segregation ▼

Es mejor tener varias interfaces específicas para cada cliente que una única interfaz de propósitos generales.

D Dependency Inversion ▼

Lo concreto debe depender de lo abstracto, nunca al revés. Los módulos de alto y bajo nivel deben depender de abstracciones.

Relaciones UML — tocá cada una

──▷ Herencia ▼

Proposición "es un". La subclase recibe todos los componentes públicos y protegidos de la superclase. Ocurre en tiempo de diseño, no de ejecución.

┄┄▷ Implementación ▼

Relación entre una clase y una interfaz: la clase implementa las capacidades que define la interfaz.

──> Asociación ▼

Objetos que colaboran entre sí. No es una relación fuerte: el tiempo de vida de un objeto no depende del otro. Puede ser unidireccional o bidireccional.

◇──> Agregación ▼

Variante de asociación, "tiene un" de tipo parte-todo. Si la clase contenedora deja de existir, las clases contenidas pueden seguir existiendo.

◆──> Composición ▼

Tipo de agregación más restrictivo: cada parte pertenece a un solo todo. Si la clase contenedora deja de existir, las contenidas también.

┄┄> De uso ▼

Una clase usa a otra como parámetro de una operación o la instancia localmente. El objeto no se guarda como atributo permanente.

Diagrama de clases vs. de secuencia

📂 Clases ▼

Vista estática, "una foto" del sistema. Muestra atributos, métodos y relaciones. Surge del análisis de responsabilidades de los objetos.

⚡ Secuencia ▼

Vista dinámica, "el diagrama de clases en movimiento". Muestra mensajes, activaciones y líneas de vida a través del tiempo. Se arma a partir de un caso de uso.

Antipatrones frecuentes — tocá cada uno

Objeto todopoderoso▼

Toda la funcionalidad se concentra en un solo objeto, en vez de repartirse en responsabilidades separadas.

Clase gorda▼

Una clase con demasiados atributos y métodos, que termina siendo responsable de lógica de negocio que no le corresponde.

Modelo de dominio anémico▼

Objetos que solo llevan datos, sin ninguna lógica de negocio propia. No es realmente orientado a objetos.

Singletonitis▼

Abuso del patrón Singleton (o de cualquier patrón) en lugares donde no es necesario ni adecuado.

Fábrica de combustible▼

Diseñar de manera innecesariamente compleja: sobre-modelar funcionalidad que no se sabe si se va a necesitar.

Gran bola de lodo▼

Construir un sistema sin ninguna estructura definida. Sin estructura, la implementación se desmorona.

Acoplamiento secuencial▼

Una clase cuyos métodos deben invocarse en un orden determinado para funcionar; si se llaman en otro orden, se rompe.

Problema del yoyó▼

Estructuras de herencia tan fragmentadas que se vuelven difíciles de seguir y entender.
